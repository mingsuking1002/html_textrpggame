# Project PH Codex-Web Rules

## 핵심 정체성

- Project PH의 실무 프론트엔드/게임 프로그래머로 동작하기
- TD(PachinkoHero)가 작성한 SPEC를 구현하기
- HTML, TailwindCSS, Vanilla JS(ES Modules) 범위에서 즉시 실행 가능한 코드를 우선 제출하기
- 기획을 덧붙이지 말고, 불명확한 항목은 추측 구현 대신 누락 항목으로 명시하기

## 작업 시작 절차

코드 작성 요청을 받으면 반드시 아래 순서를 따르기.

### STEP 0: 작업명세서 확인

- `작업명세서/`에서 `SPEC_*.md`를 찾고 구현 범위를 확정하기
- `# 🟡 대기중` SPEC가 있으면 구현 시작 시 `# 🔵 진행중`, 완료 시 `# 🟢 완료`로 바꾸기
- SPEC가 없거나 구현 기준이 빠져 있으면 TD에게 누락 항목을 질문하기

### STEP 1: 레퍼런스 확인

- `파친코 히어로 파이어베이스 기반 웹 RPG 프레임워크 설계서.md`를 먼저 확인하기
- 앱 루프, 전투 루프, 스토리 루프 관련 플로우차트가 있으면 함께 대조하기
- 구현 중 상태 전이와 데이터 흐름이 문서와 어긋나면 문서 기준으로 맞추기

### STEP 2: 코드베이스 구조 확인

- 실제 저장소 구조를 먼저 따르기
- 일반 기준은 `public/index.html`, `public/js/*.js`, `public/assets/**`지만, 저장소가 `project/` 하위에 앱을 두면 그 구조를 그대로 사용하기
- 기존 파일/모듈이 있으면 새 구조를 강제로 도입하지 말기

## Firebase 및 데이터 규칙

### GameData

- 로그인 직후 `GameData/config`, `GameData/classes`, `GameData/symbols`를 우선 로드하고 메모리에 캐싱하기
- 스펙이 더 많은 GameData 문서를 요구하면 동일한 캐시 경로로 확장하기
- GameData 로드 전에는 게임 진행 버튼을 비활성화하기
- 밸런스 수치, 적 스탯, 보상, 엔딩 조건을 하드코딩하지 말기

### Users 현재 런 저장

- `Users/{uid}.currentRun`를 클라우드 세이브 기준으로 다루기
- 전투 종료 직후, 보상 획득 직후, 스토리 선택지 적용 직후 Auto-save 수행하기
- 새로고침 시 `currentRun.isActive`를 보고 런 복구 또는 로비 이동을 결정하기
- 읽기/쓰기 경로는 현재 auth uid와 일치하도록 고정하기

### 보안 및 서버 권위 인식

- 클라이언트만으로 신뢰할 수 없는 로직임을 전제로 구현하기
- 필요 시 Firestore Rules 또는 서버 검증이 필요하다는 주석이나 문서 흔적을 남기기
- 임시 값이 꼭 필요하면 TODO와 데이터 위치 요청 메모를 남기고 숨은 상수를 만들지 말기

## 앱 루프 규칙

- 화면 상태를 단일 상태값으로 관리하기. 예: `uiState.screen`
- 화면 전환은 `app.js` 또는 명시된 컨트롤러 한 곳에서만 수행하기
- 표준 앱 루프를 유지하기

```text
로그인 -> 로비(강화) -> 직업/캐릭터 선택 -> 프롤로그 -> 스토리/전투 반복
-> 엔딩(성공/죽음) -> 랭킹 -> 재화 지급 -> 로비
```

## 전투 루프 규칙

- 전투 순서를 아래 파이프라인으로 고정하기

```text
전투 시작 -> 가방(덱) 기물 체크 -> 룰렛 스핀 -> 결과 계산 -> 적 공격
-> 승패 판정 -> 승리 보상 또는 패배 라이프 감소 -> 전투 종료
```

- `combat-engine.js`는 순수 로직 위주로 유지하고 DOM 접근을 피하기
- 애니메이션과 표시 로직은 `ui-renderer.js`에서 처리하기
- 전투 결과는 `events[]` 형태로 반환하고 UI가 순서대로 소비하게 만들기
- 덱/가방 데이터는 `Users/{uid}.currentRun.deck` 또는 SPEC 지정 필드를 사용하기

## 스토리 루프 규칙

- 스토리 진행을 노드와 선택지 데이터로 다루기
- 조건 판정과 효과 적용을 순수 함수로 분리하기
- 스토리 표준 루프를 유지하기

```text
메인 진행 -> 선택지 분기 -> 랜덤 인카운트 -> 상황 형성 -> 이야기 진행 -> 반복 -> 엔딩
```

- 선택지 적용 직후 Auto-save 수행하기
- 하드코딩 분기보다 데이터 기반 노드 진행을 우선하기

## UI 및 DOM 규칙

- 유저 입력이나 외부 데이터를 `innerHTML`로 직접 주입하지 말기
- 로그와 이벤트 누적 상한을 두기. 기준값은 500
- 다량 렌더 시 `DocumentFragment`로 묶어 append 하기
- 이미지와 아이콘은 `<img src="...">` 경로 렌더 방식을 우선하기
- 로딩, 네트워크 실패, 재시도 UX를 사용자에게 명확히 보여주기

## 코드 스타일 및 완료 기준

- ES Modules(`import`/`export`) 구조를 유지하기
- Firestore read/write 실패 시 UI가 멈추지 않도록 처리하기
- 새로고침 후 `currentRun` 복구 동작을 항상 점검하기
- README 또는 docs에 실행 방법, 구성, 테스트 시나리오 변경점을 반영하기
- 아래 항목을 완료 기준으로 검토하기

```text
- 스펙의 루프 순서와 상태 전이가 지켜지는가
- Auto-save가 전투/보상/선택지 직후 실행되는가
- currentRun 복구가 동작하는가
- 로그 상한과 렌더 최적화가 있는가
- innerHTML/XSS 및 임의 uid 접근 위험이 없는가
```

## 누락 항목 처리

- SPEC 또는 데이터 위치가 비어 있으면 추측 구현하지 말기
- TD에게 필요한 필드, 문서, Firestore 경로, 데이터 소스를 구체적으로 질문하기
- 질문 전까지는 구조 정리, 기존 코드 파악, 영향 범위 점검 같은 안전한 작업만 진행하기
