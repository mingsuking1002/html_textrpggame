# ────────────────────────────────────────────────────────────
# 2. Firebase/Firestore 구현 원칙
# ────────────────────────────────────────────────────────────
# 2-1. GameData는 1회 로드 + 캐싱
#   - 로그인 직후 `GameData/config`, `GameData/classes`, `GameData/symbols`를 가져오고
#     메모리에 캐싱합니다.
#   - UI는 GameData 로드 완료 전까지 “게임 시작”을 비활성화해야 합니다.
#
# 2-2. Users/{uid}는 클라우드 세이브(자동 저장)
#   - 전투 종료, 보상 획득 직후, 스토리 선택지 적용 직후에
#     `Users/{uid}.currentRun`을 업데이트(Auto-save)합니다.
#   - 새로고침 시 `currentRun.isActive`에 따라 런 복구/로비 이동이 되어야 합니다.
#
# 2-3. 데이터 하드코딩 금지
#   - 데미지/가중치/적 스탯/보상/엔딩 조건은 GameData 또는 spec에서 지정한 데이터로.
#   - 임시 값이 필요하면 TODO로 명시하고 TD에게 데이터 위치를 요청합니다.
#
# 2-4. 보안/권한(클라 한계)
#   - 클라에서만 신뢰하면 치팅이 가능합니다.
#   - 최소한 “서버 규칙(Firestore Rules)”을 고려한 문서/주석을 남깁니다.
#   - Users/{uid} 접근은 현재 auth uid와 일치하도록 구현합니다.

# ────────────────────────────────────────────────────────────
# 3. 앱 루프(상태 머신) 구현 규칙
# ────────────────────────────────────────────────────────────
# 화면(screen)을 단일 상태로 관리합니다 (예: `uiState.screen`).
# 상태 전환은 app.js(또는 명시된 컨트롤러) 한 곳에서만 수행합니다.
#
# Canonical App Loop:
#   로그인 → 로비(강화) → 직업/캐릭터 선택 → 프롤로그 → (스토리/전투 반복)
#   → 엔딩(성공/죽음) → (랭킹) → 재화 지급 → 로비

# ────────────────────────────────────────────────────────────
# 4. 전투 루프 구현 규칙 (룰렛)
# ────────────────────────────────────────────────────────────
# Canonical Combat Loop:
#   전투 시작 → 가방(덱) 기물 체크 → 룰렛 스핀 → 결과 계산 → 적 공격
#   → 승패 판정 → (승리: exp/보상) 또는 (패배: 라이프 감소) → 전투 종료
#
# 구현 지침:
#   - `combat-engine.js`는 순수 로직 위주(가능하면 DOM 접근 금지).
#   - UI 애니메이션/표시는 `ui-renderer.js`에서 담당.
#   - 전투 결과는 `events[]`로 반환하여 UI가 순서대로 소비.
#   - “가방/덱” 데이터는 Users currentRun의 deck(또는 spec 필드)에 존재.

# ────────────────────────────────────────────────────────────
# 5. 스토리 루프 구현 규칙 (노드/선택지)
# ────────────────────────────────────────────────────────────
# Canonical Story Loop:
#   메인 진행 → 선택지 분기 → 랜덤 인카운트 → 상황 형성 → 이야기 진행 → 반복 → 엔딩
#
# 구현 지침:
#   - 노드/선택지/조건/효과(effect)를 데이터로 다룹니다.
#   - 조건 판정은 순수 함수로 분리합니다(테스트 가능).
#   - 선택지 적용 직후 Users currentRun을 Auto-save.

# ────────────────────────────────────────────────────────────
# 6. UI/DOM 규칙 (Tailwind + 성능)
# ────────────────────────────────────────────────────────────
# - innerHTML로 유저 입력/데이터를 직접 주입 금지(XSS).
# - 로그/이벤트 누적 상한 설정(예: 500). 초과분 삭제.
# - DocumentFragment로 묶어서 append.
# - 이미지/아이콘은 `<img src="...">`로 렌더(경로는 GameData icon 사용).

# ────────────────────────────────────────────────────────────
# 7. 코드 스타일/품질
# ────────────────────────────────────────────────────────────
# - ES Modules (import/export)
# - 예외/에러는 사용자에게 ‘다시 시도’ 가능하게 표시(토스트/모달/로그)
# - 네트워크/로드 상태(loading)를 UI에 반영
# - README 또는 docs에 “실행 방법/구성/테스트 시나리오” 업데이트

# ────────────────────────────────────────────────────────────
# 8. 완료 기준(DoD) 기본 템플릿
# ────────────────────────────────────────────────────────────
# 구현 기능마다 최소 아래 항목을 만족해야 합니다:
# □ 스펙에 정의된 루프 순서/전이가 지켜진다
# □ Firestore read/write 실패 시 UI가 멈추지 않고 안내한다
# □ 전투/보상/선택지 직후 Auto-save가 수행된다
# □ 새로고침 시 currentRun 복구가 동작한다
# □ 로그/렌더 상한 및 최적화가 있다
# □ 보안 위험(innerHTML, 임의 uid 접근)이 없다
