# 시스템기획 01. 플랫폼과 메타

> 범위: `BOOT`, `AUTH`, `LOBBY`, `UPGRADE`  
> 포함 시스템: 인증, 부트/데이터 로드, 로비/프로필, 영구 강화  
> 함께 볼 문서: `ARCHITECTURE.md`, `SPEC_Phase1_AuthAndDataLoad.md`, `SPEC_Phase6_UpgradeSystem.md`, `SPEC_Phase14_LoadingOptimization.md`

---

## 1. 문서 목적

- 플레이 진입부와 메타 루프를 한 덩어리로 관리한다.
- "기획 변경 시 어느 파일과 함수를 같이 수정해야 하는지" 바로 보이게 한다.

## 2. 사용자 흐름

```text
BOOT
-> AUTH
-> Google 로그인
-> GameData/UserData 로드
-> LOBBY
-> UPGRADE 진입/복귀
-> 새 런 시작 또는 이어하기
```

## 3. 시스템별 정리

### 3.1 인증 및 세션 연결

**기획 의도**

- 구글 계정 기반으로 진입 장벽을 낮춘다.
- `Users/{uid}` 중심 저장 구조를 고정한다.

**핵심 규칙**

- 로그인 직후 유저 문서가 없으면 초기 문서를 생성한다.
- 세션이 살아 있으면 재로그인 없이 복구 흐름으로 진입한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `AUTH` |
| 관련 파일 | `public/js/firebase-init.js`, `public/js/db-manager.js`, `public/js/app.js` |
| 관련 JSON | 없음. 유저 문서(`Users/{uid}`) 중심 |
| 주요 함수 | `initFirebase`, `signIn`, `logOut`, `onAuthChange`, `loadUserData`, `handleGoogleLogin`, `restoreAuthenticatedSession` |
| Firestore | Read `Users/{uid}`, 최초 진입 시 Write `Users/{uid}` |
| 기획 변경 시 같이 볼 부분 | 로그인 방식 변경 시 `firebase-init.js`와 `db-manager.js.buildInitialUserDoc()`를 함께 수정 |

**예외 처리**

- 팝업 차단/취소 시 AUTH 화면 유지
- 유저 문서 생성 실패 시 런 시작 차단

### 3.2 부트/데이터 로드

**기획 의도**

- 첫 진입과 재접속 모두에서 로딩 체감을 낮춘다.
- 사용자가 현재 무엇을 기다리는지 볼 수 있게 한다.

**핵심 규칙**

- `GameData` 7문서와 `Users/{uid}`를 병렬 로드한다.
- 300ms 미만 완료 시 progress bar를 숨겨 깜빡임을 줄인다.
- Firestore multi-tab persistence를 먼저 시도한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `BOOT` |
| 관련 파일 | `public/js/firebase-init.js`, `public/js/db-manager.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 관련 JSON | `data/gamedata/config.json`, `classes.json`, `symbols.json`, `monsters.json`, `encounters.json`, `story.json`, `endings.json` |
| 주요 함수 | `loadGameDataWithProgress`, `loadAllDataParallel`, `updateBootProgressState`, `setBootStatus`, `setBootProgress` |
| Firestore | Read `GameData/config`, `classes`, `symbols`, `monsters`, `encounters`, `story`, `endings`, `Users/{uid}` |
| 기획 변경 시 같이 볼 부분 | 로딩 순서/문서 수 변경 시 `GAME_DATA_DOC_COUNT`, progress UI, README 확인 포인트를 함께 갱신 |

**예외 처리**

- 특정 GameData 문서 누락 시 부트 실패
- 캐시 사용 불가 환경은 네트워크 로드로 폴백

### 3.3 로비 및 프로필

**기획 의도**

- 메타 정보와 현재 런 상태를 한 화면에서 확인하게 한다.
- 닉네임 수정을 로비에서 즉시 처리한다.

**핵심 규칙**

- `currentRun.isActive`가 있으면 "이어하기" 기준으로 동작한다.
- 닉네임은 공백 정리 후 2~16자 기준으로 저장한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `LOBBY` |
| 관련 파일 | `public/js/app.js`, `public/js/ui-renderer.js`, `public/js/game-state.js` |
| 관련 JSON | `data/gamedata/config.json` 간접 참조, 나머지는 유저 문서 중심 |
| 주요 함수 | `renderLobby`, `renderLobbyState`, `handleStartRun`, `handleNicknameSave`, `normalizeNickname` |
| Firestore | Read `Users/{uid}` / Write `Users/{uid}.displayName` |
| 기획 변경 시 같이 볼 부분 | 로비 메타 항목 변경 시 `renderLobby()`와 `saveUserMeta()` 입력 필드 정의를 함께 수정 |

**예외 처리**

- 닉네임 저장 실패 시 UI 롤백
- currentRun 이상치 발견 시 새 런 시작 흐름으로 복귀

### 3.4 영구 강화

**기획 의도**

- 반복 플레이에 메타 성장을 제공한다.
- 시작 능력치 보너스를 직관적인 3축으로 제한한다.

**핵심 규칙**

- 현재 강화는 시작 HP, 시작 골드, 가방 용량 3종이다.
- 구매 즉시 Firestore에 저장하고 실패 시 롤백한다.
- 강화 효과는 다음 런 생성 시 반영한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `UPGRADE`, `LOBBY` |
| 관련 파일 | `public/js/app.js`, `public/js/ui-renderer.js`, `public/js/story-engine.js`, `public/js/db-manager.js` |
| 관련 JSON | `data/gamedata/config.json` (`upgrades`, `startHp`, `startGold`, `bagCapacity`) |
| 주요 함수 | `handleUpgrade`, `handleUpgradePurchase`, `renderUpgradeShop`, `getUpgradeDefinitions`, `createInitialRun`, `saveUserMeta` |
| Firestore | Read `GameData/config.upgrades`, `Users/{uid}.crystals`, `Users/{uid}.upgrades` / Write `Users/{uid}.crystals`, `Users/{uid}.upgrades` |
| 기획 변경 시 같이 볼 부분 | 강화 종류 추가 시 `config.json`, `renderUpgradeShop()`, `calculateUpgradeBonuses()`를 함께 수정 |

**예외 처리**

- 결정 부족 또는 최대 레벨 도달 시 구매 차단
- 저장 실패 시 재화/레벨 즉시 롤백

## 4. 이 문서를 수정할 때 체크할 항목

- 로그인/부트 플로우를 바꾸면 `AppState.BOOT`, `AUTH`, `LOBBY`, `UPGRADE` 전이 조건도 같이 봐야 한다.
- 강화 항목을 늘리면 `config.upgrades`와 새 런 생성 로직이 동시에 변경된다.
- 유저 메타 필드가 늘어나면 `buildInitialUserDoc`, `buildUserFallback`, 로비 렌더를 함께 갱신해야 한다.
