# 시스템기획 02. 런 시작과 스토리

> 범위: `CLASS_SELECT`, `PROLOGUE`, `STORY`  
> 포함 시스템: 직업 선택, 런 생성, 스토리 노드 진행, 인카운트/이벤트/상점  
> 함께 볼 문서: `ARCHITECTURE.md`, `SPEC_Phase2_StoryAndClassSelect.md`, `SPEC_Phase8_ContentExpansion.md`

---

## 1. 문서 목적

- 런이 어떻게 시작되고 어떤 데이터로 스토리가 진행되는지 정리한다.
- 신규 직업/스토리/상점 확장 시 수정 지점을 빠르게 찾을 수 있게 한다.

## 2. 사용자 흐름

```text
LOBBY
-> CLASS_SELECT
-> 직업 선택
-> currentRun 생성
-> node_prologue 진입
-> STORY
-> 선택지 / 인카운트 / 이벤트 / 상점 반복
```

## 3. 시스템별 정리

### 3.1 직업 선택 및 런 생성

**기획 의도**

- 플레이어가 런의 기본 전투 성향을 시작 시점에 선택하게 한다.
- 메타 강화 결과가 새 런에 자연스럽게 반영되게 한다.

**핵심 규칙**

- 현재 직업은 전사, 궁수, 마법사 3종이다.
- 직업 선택 시 `config + user.upgrades` 기준으로 시작 HP/골드/가방을 계산한다.
- 시작 노드는 항상 `node_prologue`다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `CLASS_SELECT`, `PROLOGUE` |
| 관련 파일 | `public/js/app.js`, `public/js/story-engine.js`, `public/js/ui-renderer.js` |
| 관련 JSON | `data/gamedata/classes.json`, `config.json` |
| 주요 함수 | `buildClassSelectionModels`, `handleClassSelect`, `createInitialRun`, `buildInitialDeck`, `renderClassSelection` |
| Firestore | Write `Users/{uid}.currentRun` |
| 기획 변경 시 같이 볼 부분 | 새 직업 추가 시 `classes`, `STARTING_DECK_RECIPES`, 클래스 이미지, 직업 카드 렌더를 함께 수정 |

**예외 처리**

- 잘못된 `classId`는 런 생성 차단
- 중복 클릭은 선택 잠금으로 방지

### 3.2 스토리 노드 진행

**기획 의도**

- 스토리를 노드와 선택지 데이터로 운영한다.
- 분기 조건과 효과를 데이터 중심으로 유지한다.

**핵심 규칙**

- 각 노드는 `title`, `text`, `type`, `choices`, `onEnter`를 가진다.
- 선택지 적용 후 `flags`, `gold`, `hp`, `stage`, `currentNodeId`가 갱신된다.
- 스토리 선택 직후 auto-save가 걸린다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `PROLOGUE`, `STORY` |
| 관련 파일 | `public/js/story-engine.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 관련 JSON | `data/gamedata/story.json` |
| 주요 함수 | `loadNode`, `applyChoice`, `applyEffects`, `meetsConditions`, `enterStoryNode`, `presentStoryNode`, `handleStoryChoice` |
| Firestore | Write `Users/{uid}.currentRun` |
| 기획 변경 시 같이 볼 부분 | 스토리 효과 키를 추가하면 `applyEffects()`와 데이터 문서를 동시에 확장해야 한다 |

**예외 처리**

- 잘못된 `nextNodeId`는 안전 복귀 처리 필요
- 조건 미충족 선택지는 노출/활성 정책을 명확히 유지해야 함

### 3.3 인카운트와 이벤트

**기획 의도**

- 동일 루트에서도 매번 다른 전개를 제공한다.
- 전투, 보상, 이벤트, 상점을 하나의 인카운트 시스템으로 묶는다.

**핵심 규칙**

- 인카운트는 가중치 기반 랜덤 추첨이다.
- `minStage`, `maxStage`, `requiredFlags`를 만족해야 후보가 된다.
- 결과 타입은 `combat`, `reward`, `event` 중심으로 사용한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `STORY`, `COMBAT` 진입 직전 |
| 관련 파일 | `public/js/story-engine.js`, `public/js/app.js` |
| 관련 JSON | `data/gamedata/encounters.json`, `story.json`, `monsters.json` |
| 주요 함수 | `rollEncounter`, `applyRewardEncounter`, `handleEncounterTrigger`, `pickEncounterMonsterId`, `advanceStage` |
| Firestore | reward/event/combat 분기 후 `currentRun` 저장 |
| 기획 변경 시 같이 볼 부분 | 인카운트 타입 추가 시 `handleEncounterTrigger()` 분기와 데이터 스키마를 함께 변경 |

**예외 처리**

- 유효 후보가 없으면 후속 노드로 안전 진행
- 전투형 인카운트는 몬스터 ID 무결성 확인 필요

### 3.4 상점 시스템

**기획 의도**

- 스토리 도중 골드 사용처를 제공해 선택의 의미를 만든다.
- 덱 성장을 스토리 루프 내부에서도 가능하게 한다.

**핵심 규칙**

- 상점 아이템은 `story.shopItems[]`로 정의한다.
- 구매 시 골드를 차감하고, 빈 슬롯이 있으면 덱에 기물을 추가한다.
- 구매 직후 auto-save가 실행된다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `STORY` 내 shop 노드 |
| 관련 파일 | `public/js/ui-renderer.js`, `public/js/app.js`, `public/js/story-engine.js` |
| 관련 JSON | `data/gamedata/story.json`, `symbols.json` |
| 주요 함수 | `renderStory`, `handleShopPurchase`, `addSymbolToDeck`, `queueAutoSave` |
| Firestore | Write `Users/{uid}.currentRun.gold`, `deck` |
| 기획 변경 시 같이 볼 부분 | 상점 가격 정책을 바꾸면 `story.json` 데이터와 구매 실패 UX 메시지를 함께 조정 |

**예외 처리**

- 골드 부족 시 구매 차단
- 덱이 가득 찼을 때 지급 정책을 유지/변경할지 명확히 해야 함

## 4. 이 문서를 수정할 때 체크할 항목

- 스토리 노드 ID를 바꾸면 `afterEncounter`, `storyNodeId`, 엔딩/복구 노드 참조가 같이 깨질 수 있다.
- 직업 추가는 전투 시작 덱과 클래스 카드, 이미지 에셋까지 한 번에 봐야 한다.
- 스토리 효과가 늘어나면 `story-engine.js`의 순수 함수 범위를 유지해야 한다.
