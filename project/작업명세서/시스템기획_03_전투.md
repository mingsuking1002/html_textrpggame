> 범위: `COMBAT`  
> 포함 시스템: 전투 엔진, 전투 UX, 리롤, 시너지, 전리품/덱 성장  
> 함께 볼 문서: `ARCHITECTURE.md`, `SPEC_Phase3_CombatEngine.md`, `SPEC_Phase7_CombatUI.md`, `SPEC_Phase9_RerollAndCrystalEconomy.md`, `SPEC_Phase10_SynergySystem.md`

## 1. 문서 목적

- 전투의 순수 로직과 UI 연출을 분리해서 이해할 수 있게 한다.
- 밸런스 변경 시 데이터 수정과 코드 수정의 경계가 어디인지 명확히 한다.

## 2. 사용자 흐름

```text
STORY / ENCOUNTER
-> COMBAT 진입
-> preview spin
-> 리롤 여부 결정
-> 결과 확정
-> 시너지 계산
-> 데미지/회복/적 공격
-> 승패 판정
-> 보상/후속 노드 이동
```

## 3. 시스템별 정리

### 3.1 전투 엔진

**기획 의도**

- 덱 구성과 난수 결과가 전투 성과를 직접 좌우하도록 만든다.
- 전투 로직은 DOM/Firebase와 분리된 순수 계산으로 유지한다.

**핵심 규칙**

- 덱 기반 `spin` 결과에서 공격/방어/회복 합계를 계산한다.
- 적 공격은 플레이어 행동 이후 수행한다.
- 피해 공식은 `config.armorConstant`를 사용한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `COMBAT` |
| 관련 파일 | `public/js/combat-engine.js`, `public/js/app.js` |
| 주요 함수 | `buildCombatEnemy`, `createCombatState`, `spin`, `executeCombatRound`, `calculateDamage` |
| 데이터 입력 | `GameData/symbols`, `GameData/monsters`, `GameData/config.armorConstant`, `currentRun.deck` |
| 기획 변경 시 같이 볼 부분 | 데미지 공식을 바꾸면 `config` 수치와 `calculateDamage()`를 함께 검토 |

**예외 처리**

- 존재하지 않는 몬스터 ID는 전투 시작 차단
- 비정상 spin 결과는 정규화 후 처리

### 3.2 전투 UX

**기획 의도**

- 룰렛 결과를 "즉시 숫자 처리"가 아니라 보는 재미가 있는 전투로 만든다.
- 로그와 숫자 팝업으로 결과를 직관적으로 전달한다.

**핵심 규칙**

- preview 생성 후 확정 단계가 분리돼 있다.
- 슬롯 공개, 데미지 숫자, 적 HP bar, 로그가 순서대로 갱신된다.
- 전투 로그는 500개 상한을 유지한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `COMBAT` |
| 관련 파일 | `public/js/ui-renderer.js`, `public/js/app.js` |
| 주요 함수 | `renderCombatScreen`, `renderCombatRoundResult`, `renderCombatVictory`, `renderCombatDefeat`, `createLogBuffer` |
| 상태 연결 | `combatState.lastSpinResult`, `turnCount`, `enemy.currentHp`, `currentRun.hp` |
| 기획 변경 시 같이 볼 부분 | 전투 HUD 항목을 바꾸면 `renderCombatScreen()`과 `buildCombatContext()`를 같이 봐야 함 |

**예외 처리**

- 연출 중 중복 입력 차단
- 렌더 도중 오류 시 현재 combatState 기반 재렌더 가능해야 함

### 3.3 리롤 및 시너지

**기획 의도**

- 한 턴 안에 비용을 지불하고 결과를 다시 노리는 선택을 제공한다.
- 같은 타입 기물이 모였을 때 추가 효율을 제공한다.

**핵심 규칙**

- 리롤은 현재 25골드를 소비한다.
- 시너지는 공격/방어/회복 3종이며 최소 2개부터 발동한다.
- preview 상태도 저장/복구 가능해야 한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `COMBAT` |
| 관련 파일 | `public/js/combat-engine.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 주요 함수 | `calculateSynergies`, `createCombatSpinPreview`, `handleCombatSpin`, `handleCombatReroll`, `showRerollOption` |
| 데이터 입력 | `GameData/config.rerollCost`, `GameData/config.synergies` |
| 기획 변경 시 같이 볼 부분 | 시너지 룰 변경 시 `config.synergies`와 전투 요약 UI를 함께 수정 |

**예외 처리**

- 골드 부족 시 리롤 차단
- preview 생성 후 새로고침 시 `combatContext` 복구 필요

### 3.4 전리품과 덱 성장

**기획 의도**

- 전투 승리가 즉시 다음 전투의 성능 향상으로 이어지게 한다.
- 보상과 덱 구성이 전투 시스템 내부에서 닫히지 않고 스토리 루프와 연결되게 한다.

**핵심 규칙**

- 몬스터는 골드와 확률형 전리품을 준다.
- 보상 기물은 빈 슬롯에만 추가된다.
- 보상 결과는 토스트와 승리 연출에 표시된다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `COMBAT`, `STORY` 복귀 직후 |
| 관련 파일 | `public/js/combat-engine.js`, `public/js/story-engine.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 주요 함수 | `applyMonsterRewards`, `addSymbolToDeck`, `formatCombatRewards`, `renderCombatVictory`, `resolveCombatRound` |
| 데이터 입력 | `monsters[*].goldReward`, `monsters[*].lootTable`, `currentRun.deck` |
| 기획 변경 시 같이 볼 부분 | 드랍 구조를 바꾸면 `monsters.json`과 전투 승리 메시지/덱 추가 실패 정책을 같이 수정 |

**예외 처리**

- 덱이 가득 차면 기물 지급 실패 가능
- 회복/골드 수치는 음수 방지와 clamp 유지 필요

## 4. 이 문서를 수정할 때 체크할 항목

- 전투 수치 조정은 가능하면 `GameData` 우선, 코드 상수는 최소화한다.
- UI 연출을 바꾸더라도 `combat-engine.js`는 순수 함수 구조를 유지한다.
- 전투 중 저장/복구를 깨지 않으려면 `combatContext` 필드와 함께 검토해야 한다.
