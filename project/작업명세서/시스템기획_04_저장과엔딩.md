# 시스템기획 04. 저장과 엔딩

> 범위: `currentRun` 저장/복구, `ENDING_SUCCESS`, `ENDING_DEATH`, `RANKING`, `PAYOUT`  
> 포함 시스템: 클라우드 세이브, 로컬 백업, 런 복구, 엔딩, 정산, 랭킹  
> 함께 볼 문서: `ARCHITECTURE.md`, `SPEC_Phase4_CloudSave.md`, `SPEC_Phase5_EndingAndRanking.md`, `SPEC_Phase13_QualityAndExtensibility.md`

---

## 1. 문서 목적

- 웹 환경에서 가장 중요한 "런 손실 방지"와 "종결 보상" 구조를 묶어 관리한다.
- 저장 시점, 복구 경로, 엔딩 정산을 하나의 생명주기로 이해할 수 있게 한다.

## 2. 사용자 흐름

```text
런 진행 중 auto-save
-> 새로고침/재접속
-> active currentRun 검사
-> STORY 또는 COMBAT 복구
-> 엔딩 조건 충족
-> 정산 및 메타 반영
-> 랭킹 등록
-> LOBBY 복귀
```

## 3. 시스템별 정리

### 3.1 클라우드 세이브와 로컬 백업

**기획 의도**

- 주요 진행 시점을 놓치지 않고 저장한다.
- Firestore 저장 실패 시에도 최소한의 복구 수단을 남긴다.

**핵심 규칙**

- 저장 트리거: 런 시작, 선택지 적용, 상점 구매, 전투 진입, 라운드 종료, 전투 종료, 엔딩 진입.
- 저장 실패 시 `localStorage`에 동일 스냅샷을 미러링한다.
- `Users/{uid}.currentRun`이 클라우드 기준 원본이다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | 전 화면 공통 |
| 관련 파일 | `public/js/app.js`, `public/js/db-manager.js`, `public/js/story-engine.js` |
| 주요 함수 | `persistRun`, `queueAutoSave`, `flushQueuedAutoSave`, `saveCurrentRun`, `saveLocalBackup`, `loadLocalBackup` |
| Firestore | Write `Users/{uid}.currentRun` |
| 기획 변경 시 같이 볼 부분 | 저장 트리거를 추가/제거하면 `queueAutoSave()` 호출 지점과 QA 체크리스트를 함께 수정 |

**예외 처리**

- Firestore 쓰기 실패 시 local backup 저장 후 에러 표시
- 저장 스냅샷은 `normalizeRunState()`로 정규화

### 3.2 런 복구

**기획 의도**

- 로그인 직후 플레이어를 가능한 한 이전 상태로 되돌린다.
- 전투 중 종료된 경우도 적 HP와 preview 상태까지 복구한다.

**핵심 규칙**

- `currentRun.isActive`가 복구 기준이다.
- `combatContext.monsterId`가 있으면 COMBAT 복구를 우선 시도한다.
- 복구 불가 시 안전한 스토리 노드 또는 로비로 폴백한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `STORY`, `COMBAT`, `LOBBY` |
| 관련 파일 | `public/js/app.js`, `public/js/story-engine.js`, `public/js/combat-engine.js` |
| 주요 함수 | `restoreAuthenticatedSession`, `restoreActiveRun`, `restoreCombatFromRun`, `resolveRestoreNodeId`, `buildCombatContext`, `normalizeCombatContext` |
| 저장 필드 | `currentNodeId`, `blockedReason`, `combatContext.*` |
| 기획 변경 시 같이 볼 부분 | 복구 가능한 전투 상태를 바꾸면 `combatContext` 스키마와 복구 로직을 같이 수정 |

**예외 처리**

- 잘못된 노드 참조는 안전 노드 계산 필요
- 몬스터 데이터 누락 시 COMBAT 복구 중단 후 STORY/LOBBY 복귀

### 3.3 엔딩과 정산

**기획 의도**

- 런 종료를 명확한 서사와 보상으로 마무리한다.
- 성공/실패 모두 메타 루프와 연결한다.

**핵심 규칙**

- HP 0은 기본적으로 실패 엔딩 후보가 된다.
- 특정 플래그 달성 시 성공 엔딩을 우선 판정한다.
- `currentRun.isActive`를 false로 저장한 뒤 메타를 갱신한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `ENDING_SUCCESS`, `ENDING_DEATH`, `PAYOUT` |
| 관련 파일 | `public/js/story-engine.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 주요 함수 | `checkEnding`, `resolveEndingId`, `calculateEndingOutcome`, `calculateCrystalReward`, `finalizeEnding`, `renderEndingView` |
| Firestore | Write `Users/{uid}.currentRun.isActive`, `totalGoldEarned`, `highestStage`, `crystals` |
| 기획 변경 시 같이 볼 부분 | 엔딩 배수/보상 공식을 바꾸면 `endings.json`, `calculateEndingOutcome()`, `calculateCrystalReward()`를 함께 수정 |

**예외 처리**

- 런 종료 저장 실패 시 정산 중단
- 메타 저장 실패 시 엔딩 화면에서 재시도 필요

### 3.4 랭킹

**기획 의도**

- 성공 엔딩 이후 비교 가능한 성과 지표를 제공한다.
- 높은 정산 금액과 루트 공략을 목표로 삼게 한다.

**핵심 규칙**

- `isRankable=true`인 엔딩만 랭킹 등록 대상이다.
- 랭킹은 `payout` 내림차순 상위 10개를 조회한다.
- 랭킹 실패는 보상 실패로 이어지지 않는다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | `RANKING` |
| 관련 파일 | `public/js/app.js`, `public/js/db-manager.js`, `public/js/ui-renderer.js` |
| 주요 함수 | `submitRanking`, `loadTopRankings`, `handleEndingPrimary`, `handleEndingSecondary`, `renderEndingView` |
| Firestore | Create `Rankings/*`, Read top 10 rankings |
| 기획 변경 시 같이 볼 부분 | 랭킹 정렬 기준 변경 시 Firestore 인덱스와 쿼리 조건을 함께 검토 |

**예외 처리**

- 랭킹 등록 실패: 보상은 유지하고 메시지만 노출
- 랭킹 조회 실패: 엔딩 흐름은 유지하고 빈 목록 처리

## 4. 이 문서를 수정할 때 체크할 항목

- currentRun 필드가 늘어나면 저장/복구/정규화 세 군데를 함께 수정해야 한다.
- 엔딩 공식을 바꾸면 랭킹 값과 메타 보상도 같이 바뀌는지 확인해야 한다.
- 복구 안정성이 중요한 영역이라 UI보다 상태 스키마를 먼저 합의하는 편이 안전하다.
