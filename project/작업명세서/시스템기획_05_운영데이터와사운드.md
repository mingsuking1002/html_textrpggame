# 시스템기획 05. 운영 데이터와 사운드

> 범위: GameData 운영, 에셋, Firestore Rules, 사운드/BGM  
> 포함 시스템: 사운드, 데이터 파이프라인, 이미지/사운드 에셋, 운영 보안  
> 함께 볼 문서: `ARCHITECTURE.md`, `SPEC_Phase11_SoundBGM.md`, `SPEC_Phase12_ImageAssets.md`, `SPEC_Phase13_QualityAndExtensibility.md`

---

## 1. 문서 목적

- 게임의 콘텐츠 원본과 운영 기반 시스템을 분리해서 관리한다.
- 사운드/에셋/보안 규칙처럼 "게임은 돌아가지만 놓치기 쉬운 영역"을 명확히 정리한다.

## 2. 사용자 흐름

```text
운영 데이터 준비
-> GameData 반영
-> 클라이언트 로드
-> 화면 상태별 사운드 재생
-> Firestore Rules 기준 접근 제어
-> 운영 스크립트로 검증/업로드
```

## 3. 시스템별 정리

### 3.1 사운드/BGM

**기획 의도**

- 화면과 전투 상태에 맞는 감정선을 오디오로 보강한다.
- 사용자가 볼륨과 음소거를 직접 제어할 수 있게 한다.

**핵심 규칙**

- 화면별 BGM은 `title`, `lobby`, `battle`, `ending` 4트랙이다.
- 효과음은 `spin`, `hit`, `heal`, `victory`, `defeat`, `purchase`, `click`을 사용한다.
- 볼륨과 음소거는 `localStorage`에 저장한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 화면 | 전 화면 공통 |
| 관련 파일 | `public/js/sound-manager.js`, `public/js/app.js`, `public/js/ui-renderer.js` |
| 관련 JSON | `data/gamedata/config.json` (`sounds`) |
| 주요 함수 | `initSoundManager`, `playBGM`, `playSFX`, `setVolume`, `setMuted`, `renderSoundControls`, `getBgmTrackForScreen` |
| 데이터 입력 | `GameData/config.sounds` |
| 기획 변경 시 같이 볼 부분 | 트랙 추가 시 `config.sounds`, `getBgmTrackForScreen()`, 전역 사운드 UI를 같이 수정 |

**예외 처리**

- autoplay 제한 시 첫 인터랙션 이후 unlock
- 탭 비활성화 시 BGM pause/resume

### 3.2 데이터 파이프라인

**기획 의도**

- 밸런스와 콘텐츠를 코드 하드코딩에서 분리한다.
- 운영자가 스크립트와 시트를 통해 데이터 품질을 관리할 수 있게 한다.

**핵심 규칙**

- 런타임은 `GameData` 7문서를 기준으로 동작한다.
- 검증/업로드/시트 연동 스크립트는 `scripts/` 하위에 둔다.
- 문서 간 참조 무결성이 중요하다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 파일 | `public/js/db-manager.js`, `data/gamedata/*`, `scripts/*` |
| 관련 JSON | `data/gamedata/config.json`, `classes.json`, `symbols.json`, `monsters.json`, `encounters.json`, `story.json`, `endings.json` |
| 주요 함수/스크립트 | `loadGameData`, `loadGameDataWithProgress`, `validate-gamedata.js`, `upload-gamedata.js`, `google-sheets-*.js` |
| 데이터 문서 | `config`, `classes`, `symbols`, `monsters`, `encounters`, `story`, `endings` |
| 기획 변경 시 같이 볼 부분 | 새 데이터 문서 추가 시 `GAME_DATA_DOC_IDS`, progress 계산, README 설명을 함께 수정 |

**예외 처리**

- 문서 누락 또는 참조 불일치 시 런타임 오류 가능
- 업로드 스크립트는 네트워크/권한 전제가 필요

### 3.3 이미지/사운드 에셋

**기획 의도**

- 이모지나 텍스트 대신 실제 에셋 경로를 기준으로 UI를 구성한다.
- 에셋 교체를 코드 수정 없이 데이터/파일 교체로 처리한다.

**핵심 규칙**

- 에셋은 `public/assets/images`, `public/assets/sounds` 기준으로 정적 서빙한다.
- 현재 이미지 36개, 사운드 11개 placeholder를 포함한다.
- 아이콘/이미지 경로는 데이터 문서에서 참조한다.

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 파일 | `public/index.html`, `public/js/ui-renderer.js`, `public/assets/**`, `data/gamedata/*.json` |
| 관련 JSON | `classes.json`, `symbols.json`, `monsters.json`, `endings.json`, `config.json` |
| 주요 함수 | `createIcon`, `createIconFallback`, `renderClassSelection`, `renderStory`, `renderCombatScreen` |
| 데이터 입력 | `classes[*].icon`, `symbols[*].icon`, `monsters[*].icon`, `endings[*].icon` |
| 기획 변경 시 같이 볼 부분 | 이미지 필드 명세가 바뀌면 데이터 문서와 `createIcon()` 사용 지점을 함께 확인 |

**예외 처리**

- 이미지 로드 실패 시 fallback UI 사용
- 잘못된 경로는 렌더는 되지만 시각 품질 저하 발생

### 3.4 운영 보안과 권한

**기획 의도**

- 공개 데이터와 개인 저장 데이터의 접근 범위를 최소한으로 분리한다.
- 클라이언트 권위 구조의 한계를 문서화하고 후속 보강 지점을 남긴다.

**핵심 규칙**

- `GameData`는 public read / write 금지
- `Users/{uid}`는 auth uid 본인만 read/write
- `Rankings`는 public read, 로그인 유저 create만 허용

**구현 연결점**

| 구분 | 내용 |
|------|------|
| 관련 파일 | `firestore.rules`, `public/js/db-manager.js`, `public/js/app.js` |
| 관련 JSON | 없음. Rules/Firestore 경로 정책 중심 |
| 주요 함수 | `saveCurrentRun`, `saveUserMeta`, `submitRanking`, `loadTopRankings` |
| Firestore Rules | `GameData`, `Users/{uid}`, `Rankings` |
| 기획 변경 시 같이 볼 부분 | 저장 경로 추가 시 rules와 db-manager API를 동시에 수정해야 함 |

**예외 처리**

- 랭킹 쿼리는 인덱스 필요 가능
- 현재 전투/정산은 클라이언트 권위라 서버 검증이 추가로 필요

## 4. 이 문서를 수정할 때 체크할 항목

- 데이터 구조를 바꾸면 반드시 로드 경로와 UI 렌더 경로가 같이 깨지는지 확인한다.
- 사운드/에셋은 "파일 교체만으로 운영 가능"한 구조를 유지하는 편이 좋다.
- Rules 변경은 기능 구현보다 늦게 가면 실제 배포에서 막히기 쉽다.
