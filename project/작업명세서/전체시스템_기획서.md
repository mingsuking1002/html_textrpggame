## 0. 문서 개요

### 0.1 기획의도

- 프로젝트에 이미 구현된 기능을 "화면 단위"가 아니라 "시스템 단위"로 재정리한다.
- 신규 SPEC 작성 전, 현재 무엇이 있는지 빠르게 파악할 수 있는 기준 문서를 만든다.
- 코드/데이터/Firestore 구조를 함께 묶어서 기획, 구현, QA 간 해석 차이를 줄인다.

### 0.2 기대효과

- 신규 작업 시 기존 시스템을 중복 구현하는 리스크를 줄일 수 있다.
- 밸런스 데이터 수정 시 영향 범위를 빠르게 파악할 수 있다.
- 운영 관점에서 어떤 저장 시점과 복구 시점이 존재하는지 명확히 볼 수 있다.

### 0.3 전체 시스템 개요

```text
BOOT
  -> AUTH
  -> LOBBY <-> UPGRADE
  -> CLASS_SELECT
  -> PROLOGUE / STORY
  -> ENCOUNTER (combat / reward / event / shop)
  -> COMBAT
  -> STORY 반복
  -> ENDING_SUCCESS / ENDING_DEATH
  -> RANKING / PAYOUT
  -> LOBBY

추가 흐름:
- 로그인 직후 active currentRun 존재 시 STORY 또는 COMBAT으로 복구
- 저장 실패 시 localStorage 백업으로 복구 시도
```

### 0.4 공통 용어 정리

| 용어 | 설명 |
|------|------|
| `GameData` | 게임 밸런스와 콘텐츠 원본. `config`, `classes`, `symbols`, `monsters`, `encounters`, `story`, `endings` 7개 문서로 구성 |
| `Users/{uid}` | 유저 메타 정보와 현재 런 저장 문서 |
| `currentRun` | 현재 진행 중인 1회차 플레이 정보. 직업, HP, 골드, 덱, 노드, 전투 컨텍스트 등을 포함 |
| `deck` | 전투 룰렛에 사용되는 가방/기물 목록. 빈 슬롯은 `empty` |
| `encounter` | 스토리 중간에 발생하는 랜덤 상황. 전투, 보상, 이벤트, 상점 진입을 포함 |

## 1. 인증 및 세션 연결 시스템

### 1.1 기획의도

- 구글 계정 기반으로 진입 장벽을 낮추고, UID 기준 클라우드 저장 구조를 단순화한다.
- 로그인 직후 유저 문서가 없으면 즉시 초기 문서를 생성해 런/메타 저장의 기반을 만든다.

### 1.2 기대효과

- 별도 회원가입 UI 없이 즉시 플레이 진입이 가능하다.
- 모든 저장 경로를 `Users/{uid}` 하나로 고정해 운영 복잡도를 줄인다.

### 1.3 시스템 개요

```text
BOOT -> AUTH 화면 노출 -> Google Popup Login
-> Users/{uid} 조회
-> 문서 없으면 초기 문서 생성
-> GameData/UserData 로드
-> active currentRun 여부에 따라 LOBBY 또는 런 복구 진입
```

### 1.4 관련 UI

- AUTH 화면
- "구글로 시작하기" 버튼
- 로그인 오류 메시지 영역

### 1.5 데이터

| 필드 | 타입 | 설명 |
|------|------|------|
| `Users/{uid}.displayName` | string | 로비/랭킹 표시용 닉네임 |
| `Users/{uid}.createdAt` | timestamp | 최초 가입 시각 |
| `Users/{uid}.totalGoldEarned` | number | 누적 정산 골드 |
| `Users/{uid}.highestStage` | number | 최고 도달 스테이지 |
| `Users/{uid}.crystals` | number | 메타 재화 |
| `Users/{uid}.upgrades` | object | 영구 강화 레벨 |
| `Users/{uid}.currentRun` | object | 진행 중 런 저장 |

### 1.6 예외 처리

- 팝업 차단: 브라우저 설정 안내 메시지 노출
- 로그인 취소: 취소 안내 후 AUTH 화면 유지
- 유저 문서 생성 실패: 런 시작을 막고 오류 메시지 유지

## 2. 부트/로딩 최적화 시스템

### 2.1 기획의도

- 로그인 이후 필요한 핵심 데이터를 최대한 빠르게 준비하고, 기다리는 동안 사용자가 현재 진행 상황을 알 수 있게 한다.
- Firestore 캐시를 적극 활용해 재방문 체감 속도를 개선한다.

### 2.2 기대효과

- 첫 진입과 재접속 모두에서 로딩 체감이 줄어든다.
- 어떤 데이터가 막혔는지 부트 단계에서 확인 가능해 디버깅이 쉬워진다.

### 2.3 시스템 개요

```text
Firestore multi-tab persistence 시도
-> GameData 7문서 병렬 로드
-> Users/{uid} 병렬 로드
-> progress 1/8 ~ 8/8 갱신
-> 300ms 미만 로딩은 progress bar 비노출
-> 완료 후 LOBBY 또는 런 복구
```

### 2.4 관련 UI

- BOOT 화면
- 로딩 문구
- progress bar
- 현재 로딩 중 문서 라벨

### 2.5 데이터

| 문서 | 설명 |
|------|------|
| `GameData/config` | 시작 스탯, 리롤 비용, 시너지, 사운드, 강화 정의 |
| `GameData/classes` | 직업 이름/아이콘/초기 무기군 |
| `GameData/symbols` | 기물 능력치와 아이콘 |
| `GameData/monsters` | 적 스탯/보상/전리품 |
| `GameData/encounters` | 랜덤 인카운트 정의 |
| `GameData/story` | 스토리 노드와 선택지 |
| `GameData/endings` | 엔딩 조건과 정산 배수 |
| `Users/{uid}` | 유저 메타와 currentRun |

### 2.6 예외 처리

- 특정 GameData 문서 누락: 부트 실패 처리
- 네트워크 지연: progress 라벨로 체감 대기 완화
- 캐시 비활성 환경: 네트워크 기반 로드로 폴백

## 3. 로비 및 프로필 시스템

### 3.1 기획의도

- 플레이어가 메타 진행도와 현재 런 상태를 한 화면에서 확인하고 다음 행동을 결정하도록 한다.
- 닉네임 수정 기능을 별도 화면 없이 로비에서 바로 처리한다.

### 3.2 기대효과

- "새 런 시작"과 "런 이어하기"를 명확히 분리할 수 있다.
- 로비가 메타 허브 역할을 수행해 반복 플레이 구조가 선명해진다.

### 3.3 시스템 개요

```text
로그인 완료
-> LOBBY 진입
-> 닉네임 / 누적 골드 / 최고 스테이지 / 결정 확인
-> 게임 시작 또는 런 이어하기
-> 강화 상점 이동 가능
```

### 3.4 관련 UI

- 로비 메타 카드 영역
- 닉네임 입력창 및 저장 버튼
- "게임 시작 / 이어하기" 버튼
- "강화" 버튼

### 3.5 데이터

| 필드 | 설명 |
|------|------|
| `displayName` | 로비/랭킹 공용 닉네임 |
| `totalGoldEarned` | 누적 보상 기록 |
| `highestStage` | 최고 진행도 |
| `crystals` | 강화 구매 재화 |
| `currentRun.isActive` | 이어하기 가능 여부 |

### 3.6 예외 처리

- 닉네임 제한: 공백 정리 후 2~16자 기준으로 저장
- 닉네임 저장 실패: 이전 값으로 UI 롤백
- currentRun 비정상: 새 런 시작 흐름으로 복귀

## 4. 영구 강화 시스템

### 4.1 기획의도

- 실패 후에도 메타 성장을 제공해 반복 플레이 동기를 유지한다.
- 강화 효과를 시작 HP, 시작 골드, 가방 용량으로 단순화해 이해도를 높인다.

### 4.2 기대효과

- 엔딩 후 결정 획득이 즉시 다음 런의 동기로 연결된다.
- 직업 선택 이전부터 장기 성장 루프가 형성된다.

### 4.3 시스템 개요

```text
LOBBY -> UPGRADE
-> 강화 항목 선택
-> 결정 차감 / 레벨 증가
-> Firestore 저장
-> 다음 런 생성 시 보너스 적용
```

### 4.4 관련 UI

- 강화 상점 카드 목록
- 비용, 최대 레벨, 현재 레벨 표시
- 구매 중 비활성화 처리

### 4.5 데이터

| 항목 | 현재 정의 |
|------|-----------|
| `upgrade_hp` | 시작 HP 증가, 비용 50, 최대 3레벨 |
| `upgrade_gold` | 시작 골드 증가, 비용 30, 최대 3레벨 |
| `upgrade_bag` | 가방 용량 증가, 비용 80, 최대 2레벨 |
| `Users/{uid}.upgrades` | 강화 레벨 저장 맵 |
| `Users/{uid}.crystals` | 구매에 사용하는 재화 |

### 4.6 예외 처리

- 결정 부족: 구매 차단
- 최대 레벨 도달: 버튼 비활성화
- 저장 실패: 차감한 결정과 레벨을 즉시 롤백

## 5. 직업 선택 및 런 생성 시스템

### 5.1 기획의도

- 런 시작 시 플레이 스타일을 직업 기반으로 명확히 선택하게 한다.
- 영구 강화 결과가 새 런의 시작 능력치에 즉시 반영되도록 한다.

### 5.2 기대효과

- 직업별 초기 덱 차이로 전투 감각이 달라진다.
- 강화와 런 시작이 자연스럽게 연결된다.

### 5.3 시스템 개요

```text
LOBBY -> CLASS_SELECT
-> 직업 선택
-> config + upgrades 기반 초기 스탯 계산
-> 초기 덱 생성
-> currentRun 생성 및 auto-save
-> node_prologue 진입
```

### 5.4 관련 UI

- 직업 카드 3종
- 직업 이름/이미지/대표 무기 표시

### 5.5 데이터

| 직업 | 특징 |
|------|------|
| 전사 | `sword`, `mace`, `hammer` 기반 근접 공격 시작 |
| 궁수 | `bow`, `crossbow`, `dagger` 기반 원거리 공격 시작 |
| 마법사 | `staff`, `dagger` 기반 마법 공격 시작 |

| `currentRun` 기본 필드 | 설명 |
|------------------------|------|
| `isActive` | 런 진행 여부 |
| `classId` | 선택 직업 |
| `stage` | 현재 스테이지 |
| `hp`, `maxHp` | 현재/최대 체력 |
| `gold` | 보유 골드 |
| `deck` | 가방/기물 목록 |
| `flags` | 스토리 진행 플래그 |
| `currentNodeId` | 현재 스토리 노드 |
| `encounterHistory` | 인카운트 이력 |
| `blockedReason` | 진행 제한 상태 |
| `combatContext` | 전투 복구 정보 |

### 5.6 예외 처리

- 지원하지 않는 직업 ID: 런 생성 중단
- 중복 클릭: 직업 선택 잠금 처리
- GameData 미로드 상태: 직업 선택 진입 차단

## 6. 스토리 노드 진행 시스템

### 6.1 기획의도

- 스토리를 데이터 기반 노드 구조로 관리해 분기 확장과 유지보수를 쉽게 한다.
- 선택지, 조건, 보상, 다음 노드를 코드가 아니라 데이터로 조절하게 만든다.

### 6.2 기대효과

- 신규 스토리 추가 시 `story` 문서 수정만으로 확장이 가능하다.
- 엔딩 조건과 플래그 연결이 명확해진다.

### 6.3 시스템 개요

```text
현재 노드 로드
-> title/text/type/choices 표시
-> 선택지 조건 검사
-> effects 적용
-> nextNodeId 계산
-> currentNodeId 갱신
-> 즉시 auto-save
```

### 6.4 관련 UI

- 스토리 본문 패널
- 선택지 버튼 리스트
- 현재 스테이지/HP/골드/덱 현황 카드

### 6.5 데이터

| 항목 | 설명 |
|------|------|
| `story` | 현재 21개 노드 보유 |
| 노드 타입 | `narrative`, `encounter_trigger`, `shop` 중심으로 구성 |
| 선택지 조건 | `hasFlag`, `requiredFlags`, `minStage`, `maxStage` |
| 선택지 효과 | `addFlag`, `removeFlag`, `addGold`, `heal`, `addStage`, `setStage` |

### 6.6 예외 처리

- 조건 미충족 선택지: 렌더 단계에서 비노출 또는 비활성 처리
- 잘못된 노드 참조: 안전 노드 또는 로비 복귀 필요
- 체력 0 도달: 엔딩 판정 우선 수행

## 7. 인카운트/이벤트/상점 시스템

### 7.1 기획의도

- 스토리 진행 중 랜덤성과 보상 기대를 부여해 반복 플레이의 변수를 만든다.
- 단순 전투 반복이 아니라 이벤트, 상점, 회복, 전투를 혼합해 템포를 조절한다.

### 7.2 기대효과

- 동일 루트라도 매번 다른 진행 경험을 제공한다.
- 골드 사용처와 덱 성장 선택이 자연스럽게 만들어진다.

### 7.3 시스템 개요

```text
encounter_trigger 노드 진입
-> encounterPool 기준 가중치 추첨
-> 조건(min/max stage, flags) 검사
-> 결과 타입 분기
   combat -> 전투 시작
   reward -> 즉시 보상 적용
   event -> storyNodeId 이동
   shop -> 상점 노드 표시
```

### 7.4 관련 UI

- 스토리 화면 내 인카운트 결과 진입
- 상점 판매 카드
- 상점 구매 버튼

### 7.5 데이터

| 분류 | 현재 내용 |
|------|-----------|
| 인카운트 수 | 12개 |
| 전투형 | 숲속 매복, 늑대 무리, 산적 습격, 어둠의 마법사 등 |
| 보상형 | 보물 상자, 치유의 샘, 신비한 상자 |
| 이벤트형 | 떠돌이 상인, 떠돌이 대장장이 |
| 상점 데이터 | `story.shopItems[]`에 `symbolId`, `cost` 저장 |

### 7.6 예외 처리

- 조건을 만족하는 인카운트가 없음: 후속 노드로 안전 진행
- 상점 골드 부족: 구매 차단
- 가방 가득 참: 아이템 지급 실패 처리 필요

## 8. 전투 엔진 시스템

### 8.1 기획의도

- 직관적인 룰렛/슬롯형 전투를 중심 코어로 두고, 덱 구성 결과가 매 턴 전투 결과에 직접 반영되게 한다.
- DOM과 분리된 순수 로직 모듈로 유지해 테스트와 확장을 쉽게 한다.

### 8.2 기대효과

- 기물 구성이 전투 성능과 직결되어 수집/상점/보상 시스템과 자연스럽게 연결된다.
- 엔진 로직이 UI에서 분리돼 유지보수성이 높다.

### 8.3 시스템 개요

```text
전투 시작
-> 적 생성(monsters)
-> deck 기반 spin 결과 생성
-> 공격/방어/회복/골드 합산
-> 플레이어 행동 반영
-> 적 공격 수행
-> 승리/패배 판정
-> 다음 라운드 또는 종료
```

### 8.4 관련 UI

- COMBAT 화면
- 적 정보 카드
- 플레이어 전투 HUD
- 전투 로그

### 8.5 데이터

| 항목 | 설명 |
|------|------|
| `symbols` | 현재 15종. `type`, `value`, `rarity`, `icon` 사용 |
| `monsters` | 현재 10종. `hp`, `attack`, `defense`, `goldReward`, `lootTable`, `tier` 사용 |
| `config.armorConstant` | 방어력 기반 피해 감소 계산 상수. 현재 15 |
| `combatState.enemy` | 현재 적 스냅샷 |
| `combatState.turnCount` | 진행 턴 수 |

### 8.6 예외 처리

- 잘못된 monsterId: 전투 시작 차단
- 비정상 spin 데이터: 정규화 후 처리
- 전투 중 오류: 로비 복귀 또는 안전 복구 필요

## 9. 전투 UX/리롤/시너지 시스템

### 9.1 기획의도

- 전투를 단순 계산 결과가 아니라 "기다리고 확인하는 재미"가 있는 UX로 만든다.
- 리롤과 시너지를 통해 전투 판단 요소를 추가한다.

### 9.2 기대효과

- 한 턴 안에서도 미리보기, 재선택, 결과 확인의 리듬이 생긴다.
- 동일한 덱이라도 조합 운과 리롤 판단에 따라 체감이 달라진다.

### 9.3 시스템 개요

```text
spin 버튼
-> preview 결과 생성
-> 필요 시 리롤(골드 차감)
-> 결과 확정
-> 슬롯 공개 연출
-> 시너지 계산
-> 데미지/회복 숫자 팝업
-> 로그 누적 및 승패 반영
```

### 9.4 관련 UI

- preview 슬롯 영역
- 리롤 버튼
- 시너지/토스트 메시지
- 적 HP bar 흔들림
- 데미지/회복 숫자 팝업

### 9.5 데이터

| 항목 | 현재 값 |
|------|---------|
| `config.rerollCost` | 25 골드 |
| 공격 시너지 | 공격 타입 2개 이상, 초과 1개당 +3 |
| 방어 시너지 | 방어 타입 2개 이상, 초과 1개당 +2 |
| 회복 시너지 | 회복 타입 2개 이상, 초과 1개당 +2 |
| 로그 상한 | 500개 |
| `combatContext.isAwaitingSpinCommit` | preview 확정 대기 여부 |

### 9.6 예외 처리

- 골드 부족: 리롤 차단
- 연속 클릭: resolving 중 입력 차단
- 새로고침 복구: preview 상태까지 `combatContext`로 복원

## 10. 보상/전리품/덱 성장 시스템

### 10.1 기획의도

- 전투와 인카운트가 단순 소모가 아니라 성장 루프로 이어지도록 만든다.
- 획득한 기물이 즉시 덱에 반영되어 다음 전투 체감을 바꾸게 한다.

### 10.2 기대효과

- 보상형 인카운트와 몬스터 처치 모두가 의미 있는 성장 수단이 된다.
- 스테이지가 올라갈수록 덱의 방향성이 뚜렷해진다.

### 10.3 시스템 개요

```text
전투 승리 또는 reward encounter 발생
-> 골드/회복/기물 지급
-> 빈 deck 슬롯 확인
-> 가능하면 기물 추가
-> 결과 토스트 출력
-> 다음 스토리/후속 노드 이동
```

### 10.4 관련 UI

- 전투 승리 요약
- 보상 토스트
- 덱 슬롯 변화

### 10.5 데이터

| 출처 | 설명 |
|------|------|
| `monsters[*].goldReward` | 전투 승리 골드 범위 |
| `monsters[*].lootTable` | 드랍 가능 기물과 확률 |
| `encounters[*].rewards` | 보상형 인카운트의 골드/회복/기물 |
| `currentRun.deck` | 보상 적용 대상 가방 |
| `currentRun.stage` | 인카운트 진행 시 증가하는 스테이지 |

### 10.6 예외 처리

- 빈 슬롯 없음: 기물 지급 실패 가능
- 회복 오버플로: `maxHp` 기준 clamp
- 보상 적용 중 저장 실패: currentRun 재저장 필요

## 11. 클라우드 세이브/복구 시스템

### 11.1 기획의도

- 이탈이나 새로고침이 잦은 웹 환경에서도 런 손실을 최소화한다.
- 전투 도중 종료된 경우까지 복구 가능하도록 저장 범위를 확장한다.

### 11.2 기대효과

- 모바일/데스크톱 환경에서 세션 단절 리스크가 줄어든다.
- 긴 런 구조를 웹에서도 안정적으로 운영할 수 있다.

### 11.3 시스템 개요

```text
특정 시점마다 queueAutoSave/persistRun 실행
-> Users/{uid}.currentRun 저장
-> localStorage 미러 백업
-> 로그인/새로고침 시 currentRun 검사
-> STORY 또는 COMBAT 화면 복구
```

### 11.4 관련 UI

- 사용자 직접 조작 UI는 없고, 복구 결과가 로비/스토리/전투 진입으로 반영됨

### 11.5 데이터

| 저장 트리거 | 설명 |
|-------------|------|
| 런 시작 직후 | 직업 선택 후 즉시 저장 |
| 스토리 선택지 적용 직후 | 분기 손실 방지 |
| 상점 구매 직후 | 골드/덱 변동 저장 |
| 전투 진입 직후 | 적 정보와 복귀 노드 저장 |
| 전투 라운드 종료 직후 | HP, 적 HP, 로그 저장 |
| 전투 종료 직후 | 보상 및 후속 노드 저장 |
| 엔딩 진입 직후 | 런 종료 직전 스냅샷 저장 |

| `combatContext` 필드 | 설명 |
|-----------------------|------|
| `monsterId` | 현재 전투 대상 |
| `currentEnemyHp` | 적 남은 체력 |
| `turnCount` | 현재 턴 수 |
| `logs` | 최근 전투 로그 |
| `lastSpinResult` | 마지막 preview/확정 결과 |
| `isAwaitingSpinCommit` | preview 확정 대기 상태 |
| `resumeNodeId` | 전투 종료 후 복귀 노드 |
| `restoreNodeId` | 복구 기준 노드 |
| `victoryNodeId` / `defeatNodeId` | 승패 후 이동 노드 |

### 11.6 예외 처리

- Firestore 저장 실패: localStorage 백업 저장 후 오류 표시
- currentRun 파손: `normalizeRunState`로 정규화
- 전투 복구 불가: 안전한 스토리 노드 또는 로비 복귀

## 12. 엔딩/정산/랭킹 시스템

### 12.1 기획의도

- 런의 결과를 명확한 엔딩 텍스트와 보상으로 환산해 플레이 종결감을 제공한다.
- 성공 엔딩은 랭킹과 메타 재화로 연결해 다음 플레이 동기를 유지한다.

### 12.2 기대효과

- 실패 엔딩도 결정 보상으로 메타 루프에 편입된다.

### 12.3 시스템 개요

```text
ending 조건 판정
-> endingState 생성
-> payout/crystals 계산
-> currentRun.isActive = false 저장
-> user meta(totalGoldEarned, highestStage, crystals) 저장
-> rankable ending이면 Rankings 등록
-> 상위 10위 조회 후 표시
```

### 12.4 관련 UI

- 성공/실패 엔딩 화면
- 랭킹 화면
- 정산 화면

### 12.5 데이터

| 항목 | 설명 |
|------|------|
| 엔딩 수 | 5종 |
| 성공 엔딩 | 드래곤 슬레이어, 상인의 왕, 동굴 탐험가, 리치 사냥꾼 |
| 실패 엔딩 | 여정의 끝 |
| `payout` | `round(totalGoldThisRun * payoutMultiplier + bonusGold)` |
| 결정 보상 | `base 6 + perStage 2 + successBonus 8 + rankableBonus 4` 기반 |
| 랭킹 문서 | `uid`, `displayName`, `endingId`, `stage`, `payout`, `createdAt` |

### 12.6 예외 처리

- 런 종료 저장 실패: 정산 중단
- 메타 저장 실패: 보상 지급 완료 전 재시도 요구
- 랭킹 제출 실패: 보상은 유지하고 랭킹만 실패 처리

## 14. 데이터/에셋/운영 보안 시스템

### 14.1 기획의도

- 게임 데이터를 코드와 분리해 밸런스 수정과 콘텐츠 확장을 쉽게 한다.
- 최소한의 Firestore Rules로 공개 데이터와 유저 데이터의 권한을 구분한다.

### 14.2 기대효과

- 기획 데이터 수정 시 JS 로직 변경 없이 운영이 가능하다.
- 유저 데이터는 본인 UID만 접근하도록 제한해 기본 보안선을 확보한다.

### 14.3 시스템 개요

```text
기획 데이터 작성/검증
-> GameData 반영
-> 클라이언트 로드
-> Users / Rankings는 규칙 기반 접근
-> 이미지/사운드는 public/assets 정적 서빙
```

### 14.4 관련 UI

- 직접 노출 UI보다는 전체 시스템의 기반 레이어

### 14.5 데이터

| 항목 | 설명 |
|------|------|
| Firestore Rules | `GameData`: read only / `Users/{uid}`: 본인만 read-write / `Rankings`: 전체 read, 로그인 유저 create |
| 검증 스크립트 | `scripts/validate-gamedata.js` |
| 업로드/운영 스크립트 | `upload-gamedata.js`, Google Sheets 연동 스크립트 |
| 에셋 저장소 | `public/assets/images`, `public/assets/sounds` |

### 14.6 예외 처리

- 랭킹 쿼리: `orderBy("payout", "desc")` 인덱스 필요 가능
- 현재 전투/정산 로직은 클라이언트 권위라 서버 검증 보강 필요
- GameData 누락/오참조 시 런타임 오류 가능

## 15. 후속 활용 제안

### 15.1 신규 문서 파생 추천

- `시스템별 상세 기획서`: 위 14개 시스템 중 우선순위가 높은 항목을 1개씩 분리

- 플로우차트는 텍스트 기반 요약이며, 이미지형 와이어프레임은 별도 제작 필요
- 향후 Phase 추가 시 본 문서도 함께 갱신 필요
